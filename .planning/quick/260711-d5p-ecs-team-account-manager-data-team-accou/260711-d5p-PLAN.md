<migration_and_rollback>
The migration is intentionally one-shot and ordered for recovery:

1. Create/validate the target `app/` and existing persistent `data/` directories; do not rename or copy the database.
2. Copy the new source to `app/` and the new Compose file to the root, leaving all old root source untouched.
3. Build and wait with `APP_CONTEXT=./app`; the existing `./data:/app/data` bind mount continues to point at the same host SQLite files.
4. If build, migration, startup, or health fails, stop before cleanup. The old root source remains available and can be restored with `APP_CONTEXT=. docker compose up -d --build --wait --wait-timeout 60`.
5. Only after health succeeds, delete the fixed obsolete-root list. Future deployments always use `APP_CONTEXT=./app`; rollback after cleanup is a normal redeploy of a prior Git revision into `app/`, never a database restore.

At no stage may rsync delete `data/` or `.env.production`; no migration step alters the data bind mount path. A database backup is still an operational responsibility before first deployment, because application/database migrations are independent of directory layout rollback.
</migration_and_rollback>

<threat_model>
| Threat | Risk | Mitigation |
|---|---|---|
| SSH endpoint spoofing | A runner sends source or credentials to an attacker. | Keep the verified `ECS_SSH_KNOWN_HOSTS` secret and strict host checking on every SSH and rsync operation. |
| Secret or database overwrite | A source sync replaces production credentials or SQLite state. | Exclude `.env*` and `data/`, keep the root env-file validation, and synchronize source only to app/. |
| Path/symlink escape | A crafted target redirects synchronization or cleanup outside the deployment directory. | Reject symlink deployment root/env file, use a fixed `$ECS_PATH`, explicit rsync destinations, and a literal cleanup allowlist. |
| Accidental data loss during migration | Cleanup runs before a usable replacement app exists. | Gate cleanup behind Compose health plus loopback login verification; retain root source until that point and document the immediate APP_CONTEXT=. rollback. |
| Privileged CI deployment | The deploy key can control Docker through its ECS account. | Keep only the dedicated private key in GitHub Secrets, use it in a mode-600 temporary file, and retain least GitHub permissions (`contents: read`). |
| Public HTTP exposure | The app becomes directly reachable without the TLS proxy. | Preserve `127.0.0.1:3000:3000`; the existing reverse proxy remains the public boundary. |
</threat_model>
