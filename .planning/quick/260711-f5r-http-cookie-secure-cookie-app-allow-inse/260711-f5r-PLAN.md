<objective>
Restore usable login over the public HTTP endpoint while ICP blocking prevents the configured domain from reaching HTTPS. This is an explicitly opted-in, temporary production fallback: only a literal `APP_ALLOW_INSECURE_COOKIES=true` may make the signed session cookie non-Secure.

No infrastructure, Compose, CI workflow, database, signing, rate-limit, or session-lifetime change is part of this code task.
</objective>

```bash
printf '\nAPP_ALLOW_INSECURE_COOKIES=true\n' >> /data/team-account-manager/.env.production
```

The existing GitHub Actions deployment reads that file through Compose and runs `docker compose up -d --build --wait`, which recreates the app from the pushed code and injects the flag. Do not commit this value, copy `.env.production`, alter `compose.yaml`, or expose secrets in logs. Remove the line (or set it to any value other than the exact lowercase `true`) and redeploy immediately once HTTPS works.
</external_setup>

<security_tradeoff>
This knowingly permits a session cookie over plaintext HTTP. On an untrusted network, an attacker able to observe or modify traffic can steal the signed `tam_session` cookie and impersonate the logged-in user until it expires; signing prevents forgery, not theft. The fallback does not weaken HttpOnly, SameSite=Lax, login throttling, or server-side session verification, but it is not an HTTPS substitute. Restrict use to the accepted temporary ICP-blocked deployment and remove it after domain/HTTPS availability is restored.
</security_tradeoff>
